1. 方案总体概述
“芯”鲜管家面向家庭冰箱场景的果蔬存储浪费与库存难管问题，以龙芯2K2000-i 处理器为
核心运算平台，构建一套“硬件采集-端侧AI 识别-嵌入式数据管理-云端同步-多端展示”全
链路的智能冰箱果蔬管理系统。系统在冰箱内部加装摄像头、补光LED 灯与温湿度传感器三
类感知硬件，通过周期性拍照与画面变化检测触发识别流程，结合YOLOv11n 完成果蔬种类
检测识别，并叠加新鲜度分类模型与温湿度环境数据融合判断新鲜/腐败程度，自动生成出入
库记录与保质期/新鲜度/设备异常预警，配合冰箱门外的LCD 触摸屏、Web 管理后台及移动
端实现本地与远程查询。
1.1 系统总体架构
系统采用端-边-云三层架构：
(1) 端侧（龙芯2K2000-i 硬件盒子）：负责图像采集、传感触发、AI 推理、本地数据存储与
屏幕展示，离线可独立工作。
(2) 边缘服务层：本地后端服务（FastAPI）+ 嵌入式SQLite 数据库，实现设备自主管理与
业务闭环。
(3) 云端：接收多设备上报数据，提供Web 管理后台、微信小程序与移动应用（APP）远程
查询与跨设备汇总统计。

2.3 视觉+环境双模态的硬件交互流程
冰箱内部感知硬件为摄像头、LED 灯与温湿度传感器：出入库事件的触发与判定仍以摄像头
画面为主（不再加装门磁/红外/重量传感器），温湿度传感器则按独立周期持续采集环境数
据，两路数据在后端汇合用于新鲜度修正与设备状态监控。视觉触发流程如下：
(1) 后台服务按固定周期（如每隔30 秒～1 分钟，可配置）点亮LED 补光灯并由摄像头采集
一帧图像，随后熄灭补光灯以节能。
(2) 将新采集帧与上一次采集帧进行帧间差分/背景建模比对，计算画面变化区域占比。
(3) 若变化占比超过设定阈值，判定冰箱内容物发生变动，触发完整识别流程（否则跳过本次
识别，节省算力）。
(4) 对触发帧运行YOLOv11n 完成果蔬检测与种类识别，并对每个检测框裁剪后送入新鲜度
分类模型。
(5) 对比本次与上一次识别到的各品类目标数量：数量增加判定为入库，数量减少判定为出
库，写入本地数据库并刷新触摸屏显示。
(6) 事件异步入队，待网络恢复后同步至云端。
(7) 与图像触发流程并行，温湿度传感器按独立周期采样，不受图像触发逻辑影响：
(8) 后台服务定时（如每1～5 分钟）读取SHT30/SHT31 的I2C 寄存器，获取当前温度、湿
度值。
(9) 写入本地env_log 表，并与最近的新鲜度识别结果做时间对齐，供新鲜度修正逻辑调用
（详见3.2.1 节）。
(10)若温度持续超出安全阈值（如高于8℃并持续设定时长），判定为冰箱制冷异常，生成独
立于食品新鲜度预警的设备异常告警，推送至触摸屏与云端。

3. 端侧AI 部署方案
系统在冰箱内加装SHT30/SHT31 温湿度传感器，与摄像头视觉数据共同构成“视觉+环境传
感器”双模态输入，覆盖比赛评分标准中“多模态数据采集与处理”（10 分）的要求。温湿
度数据不直接参与YOLOv11n 检测或MobileNetV3-Small 分类的模型运算，而是作为独立
特征在业务逻辑层与两级视觉AI 的输出结果进行融合，用于修正新鲜度判定与保质期预测、
识别设备制冷异常，具体作用与融合方式见3.2.1 节。
3.1 果蔬检测识别模型（YOLOv11n）
龙芯2K2000-i 无独立NPU，AI 推理需依赖CPU 软件推理，因此模型选型以轻量化、可量化
为核心原则：
•
果蔬检测识别模型：采用YOLOv11n 作为核心检测模型。相比YOLOv8n/YOLOv5-
nano，YOLOv11n 在同等参数量级下检测精度更高、推理耗时更短，且原生支持目标
检测框输出（可同时定位+识别多个果蔬），更贴合本系统识别+计数的业务需求，是
本系统端侧AI 的首选方案。
•
训练数据：公开果蔬数据集（如Fruits-360、VegFru 等）结合自建小样本数据集（实
际家庭常见果蔬多角度、多光照拍摄并标注），进行YOLOv11n 的迁移学习微调。
•
训练平台：云端/PC 端使用Ultralytics YOLOv11 官方框架（基于PyTorch）训练，
训练完成后导出为ONNX 格式。
•
模型压缩：对导出的ONNX 模型采用INT8 量化（如ONNX Runtime 的动态/静态量
化，或NCNN 的ncnn2int8 工具），降低计算量与内存占用，提升在LA364 核心上
的推理速度。

推荐模型：MobileNetV3-Small 作为新鲜度分类器（3 分类：新鲜/轻度不新鲜/腐败
变质），参数量约1.5～2.5M，单张裁剪图推理耗时可控制在数十毫秒级，与
YOLOv11n 共用同一套ONNX Runtime/NCNN 推理引擎，无需额外引入新框架。
•
备选方案：ShuffleNetV2 0.5x（比MobileNetV3-Small 更轻，适合算力更紧张场
景）或EfficientNet-Lite0（精度更高但计算量略大），可在实测阶段按精度/速度权
衡取舍。
•
训练数据：结合公开数据集（如Kaggle《Fruits Fresh and Rotten for
Classification》等，覆盖苹果、香蕉、橙子等常见果蔬的新鲜/腐败标注图像）与自建
数据（同一果蔬随存放时间推移拍摄的新鲜度渐变序列），进行迁移学习微调。
•
轻量补充策略：对训练数据覆盖不足的品类，可叠加基于HSV 颜色分布与LBP 纹理特
征的规则式新鲜度评分作为兜底/融合判据，与分类模型输出加权融合，提升覆盖面与
鲁棒性。
•
部署方式：与YOLOv11n 相同，导出ONNX 后经INT8 量化，在同一推理进程内按
先检测、后分类串行调用，避免重复加载框架。
3.2.1 温湿度数据的作用与融合方式
温湿度数据虽不直接输入AI 模型，但对新鲜度判断与设备运行监控具有重要辅助价值，主要
体现在以下几个方面：
•
新鲜度动态修正：果蔬腐败速率与环境温湿度密切相关——温度越高，微生物繁殖与
酶促褐变越快；湿度过高易滋生霉菌，过低则加速失水萎蔫。系统将produce_info
表中的静态保质期天数（shelf_life_days）与该品类入库以来的温湿度暴露历史结
合，计算动态修正系数，得到更贴近实际的“有效剩余保质期”，弥补纯视觉判断在
早期腐败（外观变化不明显）阶段的滞后性。
•
视觉+环境融合决策：将MobileNetV3-Small 输出的新鲜度等级/置信度与温湿度修
正系数做加权融合，形成综合新鲜度评分；当二者出现较大分歧（如视觉判断新鲜但
环境长期处于高温高湿）时，提高预警等级或提示用户重点关注，降低单一视觉模态
误判的风险。
•
设备异常/故障感知：温度持续偏离设定冷藏范围（如高于8℃且持续一定时长）可判
定为冰箱制冷系统异常或门体密封失效，触发独立于食品新鲜度预警的设备异常告
警，保护食品安全并提示用户及时处理，这也呼应了比赛题目中“环境感知与决策”
类AI 功能方向。

3.3 LoongArch 推理框架适配
目前主流AI 推理框架已逐步完成LoongArch 适配：
•
ONNX Runtime：微软于1.17.0 版本起正式支持LoongArch，龙芯技术团队对矩阵
乘法、卷积、转置等核心算子进行了深度向量化优化，适合作为首选推理引擎。
•
NCNN：腾讯优图推理框架已与龙芯合作完成适配，针对国产CPU 做了性能优化，适
合资源受限场景下的实时推理。
•
部署路径：YOLOv11n 与MobileNetV3-Small 训练完成后均导出为ONNX 格式，可
直接以ONNX Runtime 加载，或经NCNN 工具链（onnx2ncnn）转换
为.param/.bin 格式；对应推理引擎的LoongArch 版本可交叉编译或从龙芯官方源获
取预编译包，避免本地编译的复杂度。
•
若比赛允许外接加速硬件：可通过USB 接入轻量AI 加速棒作为算力补充，但需确认
其驱动是否支持LoongArch 主机。
3.4 端侧推理Pipeline
(1) 触发判断：周期性采集帧经帧间差分检测到显著变化后触发本次完整推理（详见2.3
节），否则跳过。
(2) 预处理：图像Resize、归一化、颜色空间转换（OpenCV，已适配LoongArch）。
(3) 检测识别：调用ONNX Runtime/NCNN 运行YOLOv11n 模型，输出果蔬类别、检测框
位置与置信度。
(4) 新鲜度分类：对每个检测框裁剪ROI，送入MobileNetV3-Small 新鲜度分类模型，输出
新鲜/轻度不新鲜/腐败变质等级与置信度。
(5) 业务判定：对比本次与上一次识别结果的品类数量变化判断入库/出库；识别或新鲜度置
信度低于阈值时转人工触摸屏确认模式。
(6) 结果写入：识别与新鲜度结果写入SQLite，同步至触摸屏与云端。
(7) 定时扫描：后台定时任务结合保质期数据与最新新鲜度分类结果，对过期或已判定腐败的
商品自动生成预警。

9. 小结
本方案以龙芯2K2000-i 为硬件核心，仅通过冰箱内置摄像头与LED 补光灯即可完成图像采
集与画面变化触发，结合YOLOv11n 检测识别与MobileNetV3-Small 新鲜度分类的两级AI
模型、嵌入式数据库、云端同步与冰箱门外触摸屏/多端展示，构建了硬件极简、软件驱动的
智能冰箱果蔬管理全链路，充分体现了LoongArch 自主架构在边缘AI 场景下的工程化落地
能力。后续建议优先完成摄像头选型安装、两级AI 模型的训练与端侧级联推理联调两项核心
工作，并对照两份原始方案文件进一步核实细节参数。
